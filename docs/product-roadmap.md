# 제품 로드맵 — MindHub Care Suite

> `docs/project-analysis.md`(현재 구조·통합 설계·기술부채)를 기반으로 한 제품 로드맵·정신과 실도입 기술요소·수익 모델.
> 작성 기준일 2026-06-24. 규제·인증·제도 사실은 공개 출처로 확인(하단 §참고). 시장 수치 등 미확인 항목은 *추정*으로 표기.
>
> ⚠️ 현재 두 시스템(MindHub·Psych-EMR)은 **가상 데이터 기반 프로토타입 데모**이며 한 DB로 미연동이다. 인증·임상검증·실서비스는 **1인 개발 범위를 넘어 의료기관·임상·법무 협력이 필요**하므로, 아래 로드맵에서 규제 항목은 "완료"가 아니라 **착수·준비·협력 게이트**로 표기한다.

## 0. 전제 — 규제 게이트가 단계를 가른다

이 제품군의 로드맵은 기능 개발 속도가 아니라 **데이터의 성격(가상→실데이터)과 규제 상태**가 단계를 결정한다.

```
가상데이터 데모  ──[Phase0 규제검토·동의설계]──▶  제한 실데이터 파일럿  ──[인증·수가·검증]──▶  실서비스
   (MVP)              핵심 게이트                     (Beta)                  장기 게이트          (Production)
```

핵심 원칙(분석 문서에서 계승):
- MindHub의 **비진단 경계**("기록·요약·전달")를 유지해야 의료기기 규제 트리거를 피한다.
- 통합의 **적색선 3종**(consent 강제를 RLS 위에서·red_flag↔동의 충돌 해소·식별자 무결성)을 실데이터 연동 *전에* 해소한다.
- 안전(위험감지)은 LLM·네트워크 비의존 규칙 기반을 유지한다.

## 1. 6개월 로드맵 (M1–M6 · 엔지니어링/제품 중심)

> 1인 개발이 **코드로 끝낼 수 있는 범위**에 집중. 규제·임상은 "착수/협력 요청"까지. project-analysis의 P0~P5 + 통합 적색선을 시간축에 배치.

| 기간 | 마일스톤 | 산출물 | 게이트/검증 |
|---|---|---|---|
| **M1** | 통합 기반: 공유 Supabase 설계 + 인증 통일 | 단일 `profiles`(role 4종), 헬퍼 통일(`app_role`/`is_doctor` 래퍼), 동명 트리거 충돌 해소안 | `0001` anon 전체읽기 정책 **완전 제거**를 차단 게이트로. mock 데모 무영향 |
| **M2** | 호스팅 실검증(미검증 부채 상환) | 호스팅 Supabase에서 RLS 격리 매트릭스·PostgREST 임베딩·Realtime 수신·청구 insert RLS 재검증 | 분석 문서 "미검증" 목록 0건화. E2E를 실DB 모드로도 1회 |
| **M3** | 식별 다리 `care_links` + 식별자 무결성(적색선 C3) | `care_links`(uuid, active 계정당 1차트 partial unique), `entries.patient_id` uuid 정규화, 연동코드 발급/입력 플로우 | 오연결·1:N 회귀 테스트. 철회 즉시 0행 |
| **M4** | ★ consent RPC(적색선 C1·H2·H4) | `get_mind_report()` — 담당의(attending)만 통과·share_* 합성·원문 미포함 전용 뷰, profiles 광역 SELECT 폐기 | consent 침투 테스트(분석 §2-1) 회귀. share별 음성테스트 |
| **M5** | ★ 안전↔동의 경계(적색선 C2) + 권한상승(H1) | red_flag "연동범위 내 항상", 미연동/철회는 앱 위기카드로(온보딩 고지), 가입 기본 role=patient·의료진 자가가입 금지 | 위험신호 경계 케이스 테스트. 가입 권한상승 시도 차단 검증 |
| **M6** | EMR `MindLogTab`(읽기전용) + 통합 데모 | 선택환자 패널 탭: mood/sleep 오버레이(출처·축방향 라벨), 복약 순응도%, 스트레스원, 위험 이력, triage 보조 배지(자동 진단/level/처방 ✗) | `api.getMindLog`+mock 폴백+Playwright. "환자 자가보고/비임상" 라벨. **통합 엔드투엔드 데모(가상데이터) 완성** |
| **M1–M6 병행** | 규제 착수(P0) | 동의서 초안, 데이터 흐름도(PIA 기초), 자문 대상(정신과 의료기관·법무) 컨택, EMR 인증제·ISMS-P 요건 갭 분석 | 코드 아닌 문서·협력. 6개월 내 "인증 취득"은 비현실 — **준비/착수까지** |

**6개월 종료 시 상태**: *가상데이터로 동작하는 완전 통합 데모* + *실데이터 연동의 안전 선결조건(적색선) 해소* + *규제·인증 준비 착수*. 아직 실환자 데이터·인증은 없음.

## 2. 1년 로드맵 (Q1–Q4 · 분기 단위)

| 분기 | 테마 | 핵심 작업 | 외부 의존(협력 필요) |
|---|---|---|---|

| **Q2** (M4–M6) | 안전·동의 경계 + 통합 데모 | §1의 M4–M6. consent RPC·안전경계·MindLogTab | — (자체 개발) |
| **Q3** | 파일럿 준비(Beta 전환) | 멀티테넌시/다(多)의사 모델, 앱수준 감사로그 전면화, 백업·암호화·세션·MFA, 동의·보관/파기 정책 구현, 비식별/가명 처리 | **의료기관 1곳 파일럿 MOU**, IRB/동의 검토, 보안 점검 |
| **Q4** | 표준화·인증 착수 | HL7 FHIR 매핑(건강정보 고속도로 연계 대비), EMR 인증제 기능성/상호운용성 자가점검, ISMS-P 준비(보안성 면제 연계), 임상 유용성 소규모 검증 설계 | **EMR 인증제 신청 준비**, ISMS-P 컨설팅, 임상 자문, (DTx 노선 택할 경우) 식약처 사전상담 |

**1년 종료 시 상태(목표)**: *제한된 실데이터로 1개 의료기관 파일럿(Beta) 운영* + *인증·표준화 착수* + *임상 유용성 초기 근거 수집 설계*. 정식 인증·수가·실서비스(Production)는 **1년 이후**(임상검증·인증 심사 기간 때문).

> 정직 단서: EMR 인증(유효 3년, 기능성·상호운용성·보안성 심사)과 DTx 허가(확증 임상시험)는 수개월~수년이 걸린다. 1인/소규모로 1년 내 "인증 취득"은 비현실적이며, 로드맵은 **신청 가능 상태 도달**을 목표로 한다.

## 3. MVP → Beta → Production 단계 정의

| 구분 | **MVP (현재)** | **Beta (파일럿)** | **Production (실서비스)** |
|---|---|---|---|

## 4. 실제 정신과 도입 시 필요한 기술 요소

> 범주별로 **[보유 / 부분 / 미구현]** + 근거. "보유"는 현재 코드에 있는 것, "부분"은 패턴은 있으나 운영 미검증, "미구현"은 새로 필요.

### 4-1. 보안·접근통제
| 요소 | 상태 | 근거/갭 |
|---|---|---|
| 역할기반 접근통제(RLS) | **보유** | `app_role`/`owns_patient` security-definer, attending 격리 |
| 데이터 경계 DB 강제(RPC) | **보유** | MindHub `get_patient_report_entries`, EMR 임베딩 |
| 암호화(전송/저장) | **부분** | HTTPS·Supabase 기본. 컬럼 암호화·키관리 미구현 |
| MFA·세션 정책 | **미구현** | 의료진 계정 다중인증 필요 |
| 감사로그(접근·변경) | **부분** | 약품 입출고 로그만. **전 임상데이터 접근/변경 로그 필요**(의료법·ISMS-P) |
| **ISMS-P 인증** | **미구현** | KISA 통합인증(관리체계16+보호대책64+개인정보21). **취득 시 EMR 인증제 보안성 점검 면제** → 우선순위 높음 |

### 4-2. 개인정보·동의
| 요소 | 상태 | 근거/갭 |
|---|---|---|
| 민감정보(건강정보) 별도 동의 | **부분** | MindHub share_* 토글·온보딩 고지. 개인정보보호법상 **명시적 동의·목적·보관기간** 문서화 필요 |
| 환자 자기결정권(공유범위) | **보유** | profiles.share_* + consent RPC(설계) — 강점 |
| 보관·파기 정책 | **미구현** | 원문 최소저장은 있으나 보관연한·파기 절차 미구현 |
| 가명·비식별 처리 | **미구현** | 2차활용·통계 시 필요 |

### 4-3. 상호운용·표준 / EMR 인증

## 5. 수익 모델

> 비진단 경계가 수익 노선을 가른다: **"건강관리용(비의료기기)"**으로 남으면 인증 부담은 작지만 수가 진입 불가, **"디지털치료기기(의료기기)"**로 가면 수가 가능성이 열리나 식약처 허가·임상시험이 전제. 두 노선을 분리해 설계한다.

### 5-1. 수익원 후보

| 수익 모델 | 대상 | 과금 | 전제조건 | 국내 현실성(근거) | 리스크 |
|---|---|---|---|---|---|
| **EMR SaaS 구독** (의사측) | 정신과 의원·클리닉 | 월 시트/기관 구독 | EMR 인증제, 안정성 | 의료 SaaS 표준 모델. EMR 인증이 신뢰·도입 조건 | 기존 EMR 교체장벽, 인증 비용 |
| **MindHub B2B2C** (병원이 환자에게 제공) | 의료기관 → 환자 | 기관 라이선스 + 환자수 | 비진단 유지 시 비의료기기로 가능 | 진입장벽 낮음(건강관리용). 케어 연속성 부가가치 | 환자 지속사용·효용 입증 필요 |
| **DTx 수가(급여)** | 처방형 디지털치료기기 | 건강보험 수가/비급여 | **식약처 허가 + 혁신의료기기 통합심사 + 건보 등재** | 가능하나 **수가 미성숙**(유사행위 부재·근거부족, 혁신의료기술 원가보상 논의 단계) | 허가·임상시험 고비용·장기 |
| **EAP(기업 임직원 정신건강)** | 기업 HR | 임직원당 구독 | 비진단·익명성 | 성장 채널(웰니스 수요) | 의료 아닌 웰니스 포지셔닝 |
| **공공/보험 제휴** | 정신건강복지센터·보험사 | 사업 계약 | 공공조달·검증 | 정책 수요 있음 | 조달·검증 사이클 길다 |
| **케어연속성 플랫폼** (통합) | 병원+환자 묶음 | 플랫폼 구독 | care_links 연동·인증 | 차별화 핵심, 장기 비전 | 통합 적색선 선결 필요 |

### 5-2. 단계별 수익 진화

```
MVP        : 무수익. 포트폴리오·데모·투자/협력 유치 자산.
Beta       : 유료 PoC/파일럿(기관 1곳). EMR SaaS 시범 구독 + MindHub B2B2C 파일럿 과금.
Production : ① 비의료기기 노선 → EMR 구독 + MindHub B2B2C(병원 라이선스)·EAP·공공.
             ② 의료기기(DTx) 노선 → 식약처 허가 후 수가/비급여 (장기·고비용, 별도 임상).
```

### 5-3. 비진단 경계가 수익에 주는 영향 (전략적 분기)

- **MindHub를 "건강관리용(비의료기기)"로 유지** → 식약처 허가 불필요, B2B2C·EAP·구독으로 빠른 수익화. 단 "치료 효과"를 광고·과금 근거로 주장하면 **즉시 의료기기 규제 트리거** → 마케팅·기능에서 비진단 경계 엄수가 곧 수익 모델 보호.
- **별도 처방형 모듈을 DTx로 분리** → 임상시험·허가를 거쳐 수가 노선. MindHub 본체(비진단 동반대화)와 **제품·법적으로 분리**해야 본체의 빠른 수익화와 DTx의 규제 경로가 충돌하지 않음.
- **Psych-EMR은 의료기기가 아닌 진료 지원 시스템** → EMR 인증제·ISMS-P가 신뢰·도입의 관문이지 식약처 허가 대상은 아님(진단·처방은 의사 행위, EMR은 기록·지원).

### 5-4. 현실 리스크 (정직)
- **DTx 수가 미성숙**: 건보 등재 가이드라인(2023.8)·혁신의료기기 통합심사는 있으나 수가 산정 기준이 확립 단계 → 수익화 시점 불확실.
- **정신과 비대면진료 제한**: 시범사업·재진/의원급 중심, 월 횟수 제한 등. "비대면 진료" 자체를 수익 핵심으로 가정하지 말 것(MindHub는 *진료 사이* 도구이지 비대면 진료가 아님 — 이 포지셔닝이 오히려 규제 회피에 유리).
- **시장 규모 수치**: 본 문서에서 단정하지 않음(*미확인*). 사업화 시 1차 시장조사 필요.
- **1인 개발 한계**: 인증·임상·영업은 팀·자본·기관 협력 전제.
